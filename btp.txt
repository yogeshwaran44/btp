an entity is a structured data model that represents a business object, similar to a table in a database. It defines fields, their data types, and relationships with other entities using CDS (Core Data Services). These entities are used to generate database tables and expose data through services like OData.

a type is a reusable data definition used to define the structure or data type of fields inside entities. It helps avoid repetition and improves consistency across the data model.

an aspect is a reusable model fragment that can be applied to multiple entities to add common fields or behaviour. It helps avoid duplication and keeps the data model clean and consistent.

a service is used to expose entities as APIs so that external applications can access and perform operations like create, read, update, and delete (CRUD). Services are typically exposed as OData or REST endpoints.

an association is used to define a relationship between two entities. It represents a link (like a foreign key) without embedding the related data.

a composition is a strong relationship between two entities where the child entity is fully dependent on the parent. If the parent is deleted, the child is also automatically deleted.

Projection is a way to create a view of an existing entity by selecting specific fields or reshaping data, usually for exposing it through a service.

CAP is a framework provided by SAP to build enterprise applications using a model-driven approach, supporting data modelling, service creation, and business logic with minimal code. Full-stack framework  Supports Java & Node.js  Auto-generates OData/REST APIs

CDS is a data modelling framework in CAP used to define entities, relationships, and services in a structured and database-agnostic way.

CDL is the textual language used to write CDS models, typically stored in .cds files.

Arrayed types in CDS are used to store multiple values of the same type in a single field, similar to a list or array.

OData (Open Data Protocol) is a standard protocol used to build and consume RESTful APIs for querying and manipulating data.
OData = a standard way to access data over HTTP  Read data  Filter data  Sort data  Expand relationships  All using URL-based queries

An OData adapter is a component that translates application data and requests into the OData protocol format, enabling standardized data access and communication.

A namespace in CDS is used to logically group and uniquely identify entities, preventing naming conflicts across different models.

Annotations in CDS are metadata used to add additional information or behaviour to entities, fields, or services without changing the core data model.


An enum (enumeration) in CDS is used to define a fixed set of constant values that a field can take.

Services:
In CAP, everything is a service.Expose data (Provided services),Consume external systems (Required services),Handle requests via event-driven architecture
Provided Services :
A CAP application mainly consists of the services it provides to clients. Such provided services are commonly declared through service definitions in CDS, and served automatically during bootstrapping
Required Services :
Required services are external services that an application depends on to function. A common example is the primary database like cds.db. Applications may also use services from other enterprise apps, microservices, or platform services such as secondary databases and message brokers.

CAP automatically manages transactions, so developers usually don’t need to handle them manually.
ACID transactions
User/tenant context
Connection pooling

Automatic Transactions (Default)
Every service call runs in a transaction
BEGIN → QUERY → COMMIT

Nested Transactions:
When one service calls another:
  await db.update(...)
  await log.insert(...)
Root transaction
Nested transactions for each service
Not fully distributed (no global atomic guarantee)



Manual Transactions (cds.tx()) - Used when you want full control

Auto-managed style (recommended):
await cds.tx(async () => {
  await INSERT.into(Authors)
  await INSERT.into(Books)
})
Automatically: Commit if success, Rollback if error

Manual control (advanced):
const tx = cds.tx()
try {
  await tx.run(...)
  await tx.commit()
} catch (e) {
  await tx.rollback(e)
}
You must handle commit/rollback manually


srv.tx() (Service-based Transaction) Same as cds.tx() but tied to a service
await srv.tx(async tx => {
  await tx.run(...)
})

Transaction Object (tx) Returned by cds.tx() or srv.tx()
Supports:
tx.run() → execute queries
tx.commit() → commit
tx.rollback() → rollback
also Contains its own tx.context

Background Jobs (cds.spawn()):
Run tasks outside current transaction
cds.spawn(async () => {
  await SELECT.from('Outbox')
})
Runs in new transaction can run once (after) repeatedly (every)

Kinds of Actions

Actions modify data in the server
Functions retrieve data
Unbound actions/functions are like plain unbound functions in JavaScript.
Bound actions/functions always receive the bound entity's primary key as implicit first argument, similar to this pointers in Java or JavaScript. The exception are bound actions to collections, which are bound against the collection and not a specific instance of the entity. An example use case are custom create actions for the SAP Fiori elements UI.




Application Router
The application router is the single point-of-entry for an application running in the Cloud Foundry environment on SAP BTP. The application router is used to serve static content, authenticate users, rewrite URLs, and forward or proxy requests to other micro services while propagating user information.
